# Scoped Browser Surface Fixtures

These fixtures are scoped-browser packet materialization inputs (`input`) paired
with the `promotion_state` and `expected_finding_kinds` the validator must
produce. They are regenerated by:

```sh
cargo run -q -p aureline-docs --bin aureline_docs_scoped_browser_surfaces -- fixture <case_name>
```

The canonical clean-Stable scoped-browser set lives in
[`artifacts/docs/m5/implement_scoped_browser_surfaces_for_docs_and_review_with_handoff_reason_return_path_and_trust_class_disclosu/support_export.json`](../../../../artifacts/docs/m5/implement_scoped_browser_surfaces_for_docs_and_review_with_handoff_reason_return_path_and_trust_class_disclosu/support_export.json);
these fixtures exercise the downgrade and blocking paths around it.

## scope_narrowed_rerun_narrowed.json

The review surface was rerun at a narrowed scope after a policy change, so the
set records a narrowing `scope_narrowed_rerun` degradation. The surfaces stay
valid and attributable but narrow to `narrowed_below_stable` instead of hiding
them — the downgrade narrows the claim, it does not hide the surface.

## out_of_bounds_scope_blocks_stable.json

The docs-reading surface is promoted to a `general_web` scope outside the
qualified docs/review/light-edit scope. A scoped browser surface may never
broaden into general web mode, so the validator blocks promotion with
`surface_scope_out_of_bounds`.

## missing_return_path_blocks_stable.json

A live-provider review handoff drops its return path. Every scoped browser
surface must stay return-path safe, so the validator blocks promotion with
`return_path_missing`.
