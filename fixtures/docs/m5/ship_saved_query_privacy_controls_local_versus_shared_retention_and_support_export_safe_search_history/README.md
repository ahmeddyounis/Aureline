# Saved-Query Privacy Fixtures

These fixtures are saved-query-privacy packet materialization inputs (`input`)
paired with the `promotion_state` and `expected_finding_kinds` the validator must
produce. They are regenerated by:

```sh
cargo run -q -p aureline-docs --bin aureline_docs_saved_query_privacy -- fixture <case_name>
```

The canonical clean-Stable saved-query set lives in
[`artifacts/docs/m5/ship_saved_query_privacy_controls_local_versus_shared_retention_and_support_export_safe_search_history/support_export.json`](../../../../artifacts/docs/m5/ship_saved_query_privacy_controls_local_versus_shared_retention_and_support_export_safe_search_history/support_export.json);
these fixtures exercise the downgrade and blocking paths around it.

## privacy_narrowed_rerun_narrowed.json

The shared-team pinned query was narrowed back to private after a policy change,
so the set records a narrowing `privacy_narrowed` degradation. The entries stay
valid and attributable but narrow to `narrowed_below_stable` instead of hiding
them — the downgrade narrows the claim, it does not hide the entry.

## retention_leak_blocks_stable.json

A private-local saved query is retained in a shared team store, exposing it to
the team beyond its owner-only privacy ceiling. Local-versus-shared retention
truth is mandatory, so the validator blocks promotion with
`retention_privacy_mismatch`.

## support_export_unsafe_blocks_stable.json

A synced history entry is marked export-safe while its redaction class still
needs redaction. A search-history entry may never travel in a support export
unless it is actually redaction-safe, so the validator blocks promotion with
`support_export_unsafe`.
