# Protected user-journey trace seeds

These fixtures are the committed reference shape for the protected
user-journey trace record — one deterministic machine-readable
export per seeded journey that conforms to
[`/schemas/traces/journey_trace.schema.json`](../../schemas/traces/journey_trace.schema.json)
and is regenerated by
[`/tools/journey_harness.sh`](../../tools/journey_harness.sh)
(wrapper) and
[`/tools/journey_harness/journey_harness.py`](../../tools/journey_harness/journey_harness.py)
(emitter).

The journey harness pairs the protected-hot-path / protected-path
vocabulary frozen in
[`/docs/benchmarks/spike_metric_names.md`](../../docs/benchmarks/spike_metric_names.md)
with the corpus manifest at
[`/fixtures/benchmarks/corpus_manifest.yaml`](../benchmarks/corpus_manifest.yaml)
so a journey trace names exactly one fixture id, exactly one
corpus-manifest revision, one minimum build-identity record, and one
degraded / fallback posture. The reserved hardware, environment,
exact-build-identity, evidence, and requirement slots are left open
so later reference-capture, release-evidence, and requirement-linkage
lanes can attach without a schema version bump. The normative
companion doc is
[`/docs/benchmarks/journey_trace_taxonomy.md`](../../docs/benchmarks/journey_trace_taxonomy.md).

## Reproducibility posture

The harness is stdlib-only Python with pinned timestamp inputs
(`SOURCE_DATE_EPOCH`, `TZ=UTC`, `LC_ALL=C`) and synthetic monotonic
ticks (`0..N`). The committed seed is stamped with
`host_os: linux` and `rustc_target_triple: unknown-linux-gnu` so
developer-local regeneration on macOS or Windows rewrites only those
two fields; every other field stays byte-stable.

`./tools/journey_harness.sh --verify-seed` re-emits every seeded
journey under a tempdir and unified-diffs against the committed
files; any drift fails loudly before the lane runs journey work.

## Regeneration

```
./tools/journey_harness.sh --emit-all --out-dir fixtures/journeys
```

To emit exactly one journey:

```
./tools/journey_harness.sh --journey journey.open_edit_save.first_useful_edit_rust_self_host \
                           --out-dir fixtures/journeys
```

## Scope rules

- Every seeded journey MUST cite a `fixture_ref` present in
  [`/fixtures/benchmarks/corpus_manifest.yaml`](../benchmarks/corpus_manifest.yaml).
  The harness validates that on emit; an unknown id is a loud failure.
- Every checkpoint and segment MUST carry a `protected_journey`
  drawn from the closed vocabulary in
  [`/schemas/traces/journey_trace.schema.json`](../../schemas/traces/journey_trace.schema.json);
  the harness never invents a synonym.
- Journey traces describe shape, not benchmark numbers. A journey
  trace is the backbone for a later benchmark-lab timing report or
  support packet; it is not itself admissible as release-evidence
  input.
- No raw file paths outside the repository, raw log bodies, or raw
  timing bytes cross the boundary. Only opaque ids, typed tokens,
  integer ticks, and short human notes.
- Fixtures that would require extra privacy, export, or retention
  review before broader CI use MUST NOT land here; the corpus-
  manifest `visibility_class: restricted` slot is reserved for
  those and they are not materialised in this repository.
