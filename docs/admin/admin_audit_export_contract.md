# Admin audit export contract

This contract freezes the tenant-scoped admin audit export packet that
administrators, support, and enterprise reviewers can rely on to answer
"who changed what, at what scope, under which policy/entitlement epoch,
and why" without treating a vendor-hosted console as the sole source of
truth.

The contract exists to keep enterprise auditability stable across
account-free local use, self-hosted deployments, mirrored sovereign
deployments, and managed convenience modes. Exports must remain useful
and explainable even when browser-only admin surfaces are unreachable or
intentionally absent.

## Scope

Frozen at this revision:

- one tenant-scoped export packet shape that can be generated by desktop
  UI, CLI/headless, self-hosted admin planes, and managed convenience
  deployments;
- a minimum export field set that preserves action class, actor class,
  scope, affected subject, policy/entitlement epochs, decision reason,
  origin truth, redaction posture, and retention window truth for every
  exported row; and
- vendor-console-independence rules that keep the contractual baseline
  reconstructable without a browser-only admin console.

Out of scope:

- implementing admin dashboards, hosted control-plane features, SIEM
  streaming, or billing systems; and
- exporting raw secrets, raw policy rule bodies, raw directory payloads,
  raw identifiers, raw hostnames, raw paths, raw tokens, or raw command
  lines.

## Export packet (`admin_audit_export_record`)

An admin audit export is a pair:

1. a human-readable summary suitable for a reviewer; and
2. a machine-readable packet conforming to
   `schemas/admin/admin_audit_export.schema.json`.

The machine-readable packet is strict and export-safe. It MUST include:

- `tenant_scope` and `policy_context`, plus the bound
  `policy_epoch_at_export` and `entitlement_epoch_at_export` (nullable but
  explicit);
- an explicit `origin_class` stating whether the export was assembled
  locally, via a self-hosted admin plane, or via managed convenience;
- `retention_window` describing the audit retention posture (default
  window versus policy-configured versus legal-hold/indefinite);
- an explicit `vendor_console_independence_class` and, when a row is not
  fully independent, an explicit `browser_handoff` note that makes the
  dependency visible rather than hidden; and
- `included_event_rows`, where every row carries the minimum field set in
  `artifacts/admin/admin_export_minimum_fields.yaml`.

## Vendor-console independence

Vendor-console independence is a declared class, not an implied promise.
Exports MUST NOT rely on a hosted console for baseline explainability.

Rules:

- A `fully_independent_local_packet` export MUST be reconstructable from
  the packet itself plus locally resolvable refs (source packets, seat
  lifecycle rows, record-state events) without any vendor-hosted admin
  surface.
- `independent_with_signed_evidence` may require additional signed
  evidence (for example, a signed continuity statement) but still MUST be
  reconstructable offline from locally available evidence.
- `vendor_console_required_for_full_detail` MUST include a typed
  `browser_handoff` note naming what detail is missing locally; the export
  must still preserve the minimum field set and remain redaction-safe.
- `vendor_console_only_acknowledged_gap` requires an explicit waiver or
  exception record referenced from the export evidence chain; a gap
  without an explicit acknowledgement is non-conforming.

## Redaction boundary

Export usefulness must not require unsafe disclosure. Exports MUST remain
reviewable without embedding:

- raw policy rule bodies, raw bundle bytes, raw signing material;
- raw issuer URLs, raw SCIM endpoints, raw mirror hostnames;
- raw IP addresses, raw device fingerprints, raw user identifiers, raw
  email or display names, raw subject claims;
- raw provider payloads, raw paths, raw tokens, raw command lines; or
- raw secret material.

Rows may carry opaque refs, typed vocabulary, redaction notes, and
reviewable summaries only.

## Change management

- Adding a new enum value or optional field is additive-minor and bumps
  `admin_audit_export_schema_version`.
- Repurposing an existing enum value, loosening the redaction boundary,
  or weakening reconstructability without a vendor console is breaking
  and requires a governance review before acceptance.
