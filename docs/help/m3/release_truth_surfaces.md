# Beta release-truth surfaces — Help / About, About card, and service health

When a reviewer or end user opens Help / About or the service-health
overview during a beta incident or evaluation, the surface MUST quote the
same beta-truth row state as the support export, the release evidence
packet, and the release notes. Forking a per-surface layout lets one entry
drift its vocabulary while another lags — a help screen claiming
"supported" while support exports quote "limited" from the same manifest
row, or a service-health chip showing "warm" freshness after the evidence
date has slipped past the review window.

This document is the reviewer-facing landing page for the beta release-
truth projection shipped under:

- `crates/aureline-shell/src/service_health/` — projects the generated M3
  governed claim manifest into a deterministic
  `ServiceHealthBetaSurface` record with per-row provenance, freshness,
  support-class, claim-posture, lifecycle, and client-scope badges.
- `crates/aureline-shell/src/about/` — projects a Help / About release-
  truth card from the same surface so the chrome quotes the same row
  vocabulary as the support export and the release-evidence packet.

Both surfaces read from
`artifacts/release/m3/claim_manifest.json` — the manifest generated by
`ci/check_m3_claim_manifest.py`. The shell never invents support,
freshness, provenance, or claim-posture truth of its own; refresh the
manifest to update what users see in-product.

Packaging, signing, update, rollback, policy, and schema-governance
truth for beta artifacts is read through
`artifacts/release/m3/artifact_graph.json` and the generated support
projection at
`artifacts/release/m3/artifact_graph_support_projection.json`. Help /
About and support export surfaces may summarize those rows, but they
quote graph node ids and generated projection fields rather than
reconstructing artifact relationships from release copy.

Install mode, channel, updater owner, durable state roots, and
exact-build install diagnostics are read through
`artifacts/release/m3/install_diagnostics/install_diagnostics_packet.json`
and the generated support projection at
`artifacts/release/m3/install_diagnostics/support_export_projection.json`.
Help / About may render this as an install card, and CLI/support export
may render it as structured diagnostics, but both quote the same
diagnostic row fingerprints.

Managed and self-serve beta rollout state is read through
`artifacts/release/m3/ring_rollout/packet.json`, the generated support
projection at
`artifacts/release/m3/ring_rollout/support_export_projection.json`, and the
state-root audit at `artifacts/release/m3/state_root_audit.md`. Help / About,
CLI, and support export may summarize rollout posture, but they must quote the
same controlled ring vocabulary and the same post-action active package state
that the headless rollout gate validates.

Update rollback state is read through
`artifacts/release/m3/update_rollback/rollback_plan.json` and the
generated support projection at
`artifacts/release/m3/update_rollback/support_export_projection.json`.
Help / About, support export, release docs, and migration surfaces quote
the same current exact build
`build-id:aureline:beta:2.1.0-beta.1:aarch64-apple-darwin:release:b7ee32adb5eb`,
rollback target `release_candidate:aureline.2_0_4_stable`, and rollback
target `exact_build_identity_ref`
`build-id:aureline:stable:2.0.4:aarch64-apple-darwin:release:1f40c9d2b4a1`.
They also quote the shared rollback vocabulary:
`retained_prior_artifact_set`, `schema_rollback_hook`,
`downgrade_eligibility_state`, and `exact_build_identity_ref`.

Correction-train state is read through
`artifacts/release/m3/correction_train/packet.json` and the generated
support projection at
`artifacts/release/m3/correction_train/support_export_projection.json`.
Help / About, release docs, support export, and release notes quote
packet id `correction.train.beta.release_control` plus the same
correction vocabulary: `correction_scope`, `correction_risk`,
`correction_evidence`, `target_channels`, `triage_lane`,
`backport_decision`, `rollback_target`, and `known_issue_update`.
The only admitted `triage_lane` values are `hotfix`, `backport`,
`correction_train_only`, and `next_cycle`; supported-line decisions use
`backport_decision` values `yes`, `no`, `defer`, or `not_applicable`.

Distributed compatibility state is read through the generated manifest index at
`artifacts/compat/m3/distributed_manifests/manifest_index.json`, the family
manifests under `artifacts/compat/m3/distributed_manifests/`, the release
packet at
`artifacts/release/m3/distributed_compatibility/release_packet.json`, and the
support export projection at
`artifacts/release/m3/distributed_compatibility/support_export_projection.json`.
Help / About, release docs, partner docs, and support exports quote the same
compatibility row refs, skew case refs, out-of-window postures, and repair
hints; unsupported windows never render as mutating support.

## What every beta row carries

Each row on the manifest minted by `ci/check_m3_claim_manifest.py`
projects into a `ServiceHealthBetaRow` carrying:

- `row_id`, `row_kind` (`canonical_claim_family` /
  `beta_surface_binding` / `beta_archetype_binding`), `headline`, and
  `claim_family` — quoted verbatim;
- `claim_posture` (declared and effective) joined to `downgraded` and
  `active_downgrade_reasons`;
- `support` (declared and effective) joined to `downgraded`,
  `target_at_beta_exit`, `target_at_stable`, and
  `downgrade_triggers_fired`;
- `lifecycle_label` (`preview` / `beta`);
- `freshness` — joins the manifest's badge (`authoritative_live`,
  `warm_cached`, `degraded_cached`, `stale`, `unverified`) with the
  derived `FreshnessStateClass` the chrome reads:
  - `current` — evidence is fresh and badged as authoritative or warm,
  - `degraded_but_current` / `unverified_but_current` — evidence is
    still inside its review window but the badge is degraded,
  - `review_due_soon` — more than half of the review window has elapsed,
  - `review_overdue` — past the review window,
  - `evidence_expired` — past twice the review window,
  - `unable_to_evaluate_freshness` — the date string could not be
    parsed or `as_of` predates the evidence date;
- `provenance` (label, evidence owner, intake / triage / release owner,
  escalation reference);
- `channel_projections` for the four channels the beta-truth lane MUST
  stay consistent on — `help_about`, `service_health`,
  `support_export`, `release_packet` — with their `binding_status`,
  `projection_kind`, `copy_field`, and `surface_ref` quoted verbatim;
- typed booleans for the user-visible failure drills:
  `claim_posture_downgraded`, `support_downgraded`, `evidence_stale`,
  `evidence_expired`, `required_projection_missing`,
  `copy_field_drifts_between_help_about_and_service_health`,
  and `honesty_marker_present` — the single bit the chrome reads when
  deciding whether to light a yellow chip on the row.

## Honesty-marker semantics

The top-level `ServiceHealthBetaSurface.honesty_marker_present` bit is
the single chip the shell chrome mirrors as a visible warning. It lights
when **any** of the following hold across **any** row:

- the row's `claim_posture.declared` does not equal
  `claim_posture.effective`;
- the row's `support.declared` does not equal `support.effective`;
- the row's derived freshness state is `review_due_soon`,
  `review_overdue`, `evidence_expired`, or
  `unable_to_evaluate_freshness`;
- one of the four required beta-truth channels
  (`help_about` / `service_health` / `support_export` /
  `release_packet`) is absent or carries
  `binding_status = "not_applicable"` while the row's effective claim
  posture is not `withdrawn`;
- the `help_about` and `service_health` channels resolve to different
  `copy_field` values for the same row.

The same bit also lights on the Help / About release-truth card so the
"Copy About card for support export" action surfaces the warning to
support exports verbatim. The chip is **never** synthesized by the
chrome itself; it only mirrors the projection.

## Client-scope chip

The projection quotes `release_channel_scope` from the manifest envelope
into a `ReleaseChannelScopeClass` chip (`nightly`, `preview`, `beta`,
`stable`, `lts`). Help / About, service health, and support exports all
read the same chip so the lane never claims "Beta" on one surface while
quoting "Preview" on another.

## Failure drills

Three named drills are exercised by fixtures under
`fixtures/release/beta_truth_cases/`:

| Fixture | Scenario | Expected surface state |
|---|---|---|
| `protected_walk_current_manifest_snapshot.json` | Every row binds with a current evidence date inside its review window and posture `claim_bearing`. | `honesty_marker_present = false`, all summary counters at 0. |
| `failure_drill_stale_evidence.json` | All rows carry an evidence date 16+ months before the chrome's `as_of`. | `evidence_expired_row_count >= 1`, `honesty_marker_present = true`. |
| `failure_drill_downgraded_claim_row.json` | A row was declared `claim_bearing` but landed `limited` with explicit downgrade reasons. | `downgraded_claim_row_count >= 1`, row's `active_downgrade_reasons` quoted verbatim, `honesty_marker_present = true`. |

## What changes when the manifest changes

When a reviewer regenerates `claim_manifest.json` — via
`ci/check_m3_claim_manifest.py` over the canonical claim-manifest seed,
the claimed-surface register, the cohort/archetype scorecard register,
the M3 compatibility report, the cohort guardrails, and the version-skew
register — Help / About and service health update on the next run
without any manual docs edit. A downgraded row appears as a downgraded
chip; a stale row appears as an overdue chip; a missing required
projection appears as a `required_projection_missing` chip.

The projection is read-only and does not:

- mutate the manifest, the seed matrix, or any upstream evidence;
- duplicate the schema validation already performed by
  `ci/check_m3_claim_manifest.py` against
  `schemas/release/m3_claim_manifest.schema.json`;
- reach out to publish / promote / rollback / yank command rails — those
  remain owned by the release-center lane and stay reserved here.
