# Scoped Companion And Desktop-Handoff Beta Contract

The beta companion projection keeps browser and mobile surfaces narrow,
explicit, and auditable. A companion row may support review, docs, light
remote edit, session join, CI status, or incident awareness, but it may not
imply desktop parity, terminal/debugger depth, unmanaged secret entry, deep
local project editing, or full admin-console replacement.

The shell-owned implementation lives in
[`crates/aureline-shell/src/companion_handoff/mod.rs`](../../../crates/aureline-shell/src/companion_handoff/mod.rs).
It projects rows, validates scope and handoff invariants, and exports
support-safe provenance. The boundary schema is
[`schemas/ux/companion_scope_contract.schema.json`](../../../schemas/ux/companion_scope_contract.schema.json).

## Claimed Rows

The seeded page covers six companion rows:

| Workflow | Companion posture | Required truth |
| --- | --- | --- |
| Review triage | comment-capable browser companion | protected approval and broad mutation reopen on desktop |
| Docs and help | read-only browser companion | source/version and warm snapshot labels stay visible |
| Light remote edit | bounded browser companion edit | save/step-up reuses desktop command policy; native-depth editing hands off |
| Remote session join | follow-capable mobile companion | scoped control requests reuse desktop approval; broad control hands off |
| CI status | status-only mobile companion | stale snapshot is labeled; rerun or publish review hands off |
| Incident awareness | light incident mobile tooling | offline snapshot is labeled; sensitive admin action hands off |

Every row carries:

- a visible client-scope label;
- visible target identity and execution-context refs;
- visible freshness, warm, stale, or offline labels;
- read-only or desktop-handoff state;
- desktop step-up or approval reuse where authority is requested;
- device-policy and quiet-hours inheritance for companion fanout;
- provenance for the surface that showed the object, the surface that owns
  approval, and the desktop destination used for handoff.

## Authority Rules

Companion surfaces can request authority but cannot grant protected authority.
The validator rejects rows that:

- claim desktop parity or terminal/debugger/admin-console replacement;
- allow unmanaged secret entry or deep local project editing;
- render stale, warm, imported, or offline state as live;
- hide target identity or execution context;
- hide both read-only and desktop-handoff state;
- let a companion own protected approval, high-risk publish, sensitive admin,
  or broad mutation;
- skip desktop step-up or approval semantics for authority-bearing actions;
- skip device-policy, quiet-hours, dedupe, or exact-reopen fanout posture.

## Support Export

`CompanionScopeBetaSupportExport` embeds the page plus one
`CompanionScopeBetaSupportRow` per source row. Support rows preserve the
workflow token, client-scope token, target ref, freshness token, handoff
reasons, approval owner, origin surface, displayed-object surface, desktop
destination, device-policy/quiet-hours state, and reused canonical event id.
They exclude raw paths, URLs, provider payloads, secrets, approval bodies,
lock-screen payloads, and actor identities.

## Fixtures

The fixture corpus lives at
[`fixtures/ux/m3/companion_scope/`](../../../fixtures/ux/m3/companion_scope/).
It is generated by the headless inspector:

```sh
cargo run -q -p aureline-shell --bin aureline_shell_companion_scope -- page
cargo run -q -p aureline-shell --bin aureline_shell_companion_scope -- rows
cargo run -q -p aureline-shell --bin aureline_shell_companion_scope -- support-rows
cargo run -q -p aureline-shell --bin aureline_shell_companion_scope -- support-export
cargo run -q -p aureline-shell --bin aureline_shell_companion_scope -- validate
```

Negative drill fixtures cover stale-label removal and companion-owned protected
approval. Both must replay typed validator defects.
