runtime contract. The contract proves that theme packages, appearance sessions,
token overlays, imported-theme mapping reports, and OS appearance changes all

## What The Packet Proves

The packet must prove:

- the first-party theme package declares dark, light, high-contrast dark, and high-contrast light support with contrast targets;
- the active `appearance_session_record` cites one active package, one revision, one live follow-system policy, and resolved mode, contrast, accent, density, text-scale, reduced-motion, and follow-system posture;
- the live follow-system policy declares every axis and makes checkpoint or confirm requirements explicit;
- token overlays preserve inherited, overridden, deprecated, and unmapped values, with unmapped slots retained as inert fallback-chain rows;
- imported themes cannot claim full parity without a mapping report, rollback ref, visible unresolved-slot counts, syntax coverage, and protected-cue honesty checks;
- OS-triggered theme, contrast, accent, text-scale, and reduced-motion changes either apply live through the appearance session, apply behind a checkpoint, hold for explicit confirm, or disclose a reload/restart posture.

The seeded fixture currently resolves to `pass_with_disclosed_gap` because the
imported theme report intentionally keeps visible unsupported and unresolved
rows. That is acceptable because the packet denies full parity and surfaces the
gaps.

## Protected Cues

Appearance changes must preserve these cue families outside color alone:

- `trust`
- `policy_lock`
- `severity`
- `source_integrity`

Every protected cue row requires non-color cues and must remain legible in
high-contrast and forced-colors postures.
